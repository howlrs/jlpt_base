## Phase 1: Issue #22 — リフレッシュトークン機構

### アーキテクチャ

```
[Client] ──POST /api/signin──> [Backend]
  ← Set-Cookie: access_token  (JWT, 30min, HttpOnly, Secure, SameSite=Lax, Path=/api)
  ← Set-Cookie: refresh_token (opaque UUID, 14d, HttpOnly, Secure, SameSite=Lax, Path=/api/auth/refresh)

[Client] ──(通常リクエスト)──> [Backend]
  Cookie: access_token
  ← 200 OK（ステートレス検証、DB参照なし）

[Client] ──(トークン期限切れ)──> [Backend]
  ← 401 Unauthorized

[Client] ──POST /api/auth/refresh──> [Backend]
  Cookie: refresh_token
  検証: Firestoreでハッシュ照合 + 未失効確認
  ローテーション: 旧トークン失効 → 新トークンペア発行
  ← Set-Cookie: access_token (新JWT, 30min)
  ← Set-Cookie: refresh_token (新opaque UUID, 14d)

[Client] ──POST /api/auth/logout──> [Backend]
  Cookie: refresh_token
  ← 当該ユーザーの全トークンを失効
  ← Clear both cookies
```

### リフレッシュトークンローテーションと漏洩検知

```
正常フロー:
  refresh_token_v1 使用 → v1失効, refresh_token_v2 発行
  refresh_token_v2 使用 → v2失効, refresh_token_v3 発行

漏洩検知:
  攻撃者が refresh_token_v1 を再利用
  → v1は既に失効済み → 再利用検出
  → token_familyに属する全トークンを即座に無効化
  → 正規ユーザーも再ログイン必要（安全側に倒す）
```

### Firestoreスキーマ

**コレクション: `refresh_tokens`**

```rust
struct RefreshToken {
    token_hash: String,        // SHA-256(refresh_token) — ドキュメントID
    user_id: String,
    token_family: String,      // UUID — ローテーション追跡用
    issued_at: i64,            // Unix timestamp
    expires_at: i64,           // issued_at + 14days
    is_revoked: bool,
    revoked_at: Option<i64>,
}
```

- **ドキュメントID**: `SHA-256(refresh_token)` — 検索用
- **token_family**: ログイン時に生成するUUID。同一ファミリー内のトークンを一括無効化できる
- **インデックス**: `user_id` + `is_revoked` の複合インデックス（ログアウト時の全無効化用）

**Claims構造体の変更:**

```rust
// 現在
pub struct Claims {
    pub sub: String,      // user_id
    pub email: String,
    pub exp: usize,       // 24h
    pub role: String,
}

// 変更後
pub struct Claims {
    pub sub: String,      // user_id
    pub email: String,
    pub exp: usize,       // 30min (1800秒)
    pub role: String,
    pub jti: String,      // UUID — トークン識別子
}
```

**リフレッシュエンドポイント:**

```rust
// POST /api/auth/refresh
// 1. Cookieからrefresh_tokenを取得
// 2. SHA-256ハッシュでFirestore検索
// 3. 未失効 & 未期限切れを確認
// 4. is_revoked=trueの既存トークンが同familyにあれば → 漏洩検知 → family全無効化
// 5. 旧トークンを失効（is_revoked=true）
// 6. 同じtoken_familyで新リフレッシュトークン発行
// 7. 新アクセストークン（JWT）発行
// 8. 両方をSet-Cookie
```

### フロントエンド変更 (jlpt-app-frontend)

**透過的リフレッシュの実装:**

```typescript
// lib/api.ts または同等のfetch wrapper
// 1. レスポンスが401の場合
// 2. POST /api/auth/refresh を呼び出し（credentials: "include"）
// 3. 成功 → 元のリクエストをリトライ
// 4. 失敗 → auth状態クリア + 「セッションが切れました」モーダル表示
// 5. 同時に複数の401が発生した場合、リフレッシュは1回だけ実行（mutex）
```

**注意点:**
- リフレッシュ中の他リクエストはキューイングし、完了後に再送
- リフレッシュ失敗時はログイン画面へのリダイレクトではなくモーダル表示（UXリサーチャー推奨）
